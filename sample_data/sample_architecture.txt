## 1. Executive Summary

This document describes the technical architecture for an AI-powered audit system designed to review delivery artifacts. The system employs a microservices-inspired architecture with a FastAPI backend, modern web frontend, and AI processing engine.

**Key Characteristics**:
- Multi-agent AI architecture
- Asynchronous task processing
- Horizontal scalability
- Secure authentication
- Cloud-native design

## 3. Technology Stack

### 3.1 Backend Stack
- **Framework**: FastAPI 0.109.0
- **Language**: Python 3.11
- **ORM**: SQLAlchemy 2.0
- **Task Queue**: Celery 5.3 + Redis 7.0
- **AI Framework**: LangChain 0.1.0
- **Authentication**: python-jose (JWT)
- **Document Processing**: PyPDF2, python-docx, docx2txt

### 3.2 Frontend Stack
- **Core**: HTML5, CSS3, JavaScript (ES6+)
- **Styling**: Custom CSS with CSS variables
- **No framework**: Vanilla JS for performance

### 3.3 Infrastructure
- **Database**: PostgreSQL 15 (production), SQLite (development)
- **Cache**: Redis 7.0
- **Web Server**: Uvicorn (development), Gunicorn + Uvicorn workers (production)
- **Reverse Proxy**: NGINX
- **Containerization**: Docker + Docker Compose
- **Cloud**: AWS (primary) / Azure (alternative)

### 3.4 External APIs
- **OpenAI API**: GPT-4 for document analysis
- **Model**: gpt-4o-mini for cost optimization

## 4. Security Architecture

### 4.1 Authentication & Authorization
- **JWT Tokens**: HS256 algorithm, 30-minute expiration
- **Password Hashing**: bcrypt with salt
- **Session Management**: Server-side validation
- **RBAC**: Role-based access control (future)

### 4.2 Data Security
- **Encryption in Transit**: TLS 1.3 (HTTPS)
- **Encryption at Rest**: Database-level encryption
- **File Storage**: Secured upload directory with restricted access
- **API Keys**: Stored in environment variables, never in code

### 4.3 API Security
- **CORS**: Configured for specific origins
- **Rate Limiting**: 100 requests/minute per user
- **Input Validation**: Pydantic models for all inputs
- **SQL Injection Protection**: Parameterized queries via SQLAlchemy

### 4.4 Security Monitoring
- Logging of all authentication attempts
- Failed login tracking
- Audit trail for document access

## 5. Scalability Design

### 5.1 Horizontal Scaling

**Web Tier**:
- Stateless FastAPI instances
- Load balanced with NGINX
- Can scale to N instances based on demand

**Worker Tier**:
- Celery workers with thread pool
- 4 concurrent threads per worker
- Auto-scaling based on queue depth

**Database Tier**:
- PostgreSQL read replicas
- Connection pooling (SQLAlchemy)
- Query optimization with indexes

### 5.2 Performance Optimization

**Caching Strategy**:
- Redis cache for user sessions
- Document metadata caching
- API response caching (future)

**Async Processing**:
- All document processing is asynchronous
- Non-blocking I/O with asyncio
- Background tasks via Celery

**Database Optimization**:
- Indexed columns: user_id, session_id, agent_type
- Batch operations where possible
- Lazy loading for relationships

## 6. Data Models

### 6.1 Core Entities

**User**:
```
- id (UUID, PK)
- username (String, unique)
- email (String, unique)
- hashed_password (String)
- created_at (DateTime)
- is_active (Boolean)
```

**AuditSession**:
```
- id (Integer, PK)
- session_id (String, unique)
- user_id (UUID, FK -> User)
- agent_type (String)
- artifact_path (String)
- checklist_path (String)
- status (Enum: pending, processing, completed, failed)
- progress_percentage (Float)
- started_at, completed_at (DateTime)
```

**AuditResult**:
```
- id (UUID, PK)
- session_id (String, FK -> AuditSession)
- compliance_score (Float)
- total_items (Integer)
- compliant_count, non_compliant_count (Integer)
- report_markdown (Text)
```

**AuditFinding**:
```
- id (UUID, PK)
- result_id (UUID, FK -> AuditResult)
- checklist_item (String)
- finding_type (Enum: compliant, non_compliant, missing)
- severity (Enum: critical, high, medium, low)
- description, recommendation (Text)
```

### 6.2 Database Schema Design

- **Normalization**: 3NF for relational integrity
- **Foreign Keys**: Enforce referential integrity
- **Cascade Deletes**: Configured for cleanup
- **Indexes**: On frequently queried fields

## Appendix A: Design Decisions

**Why FastAPI?**
- Async support for high performance
- Automatic API documentation
- Type safety with Pydantic
- Modern Python features

**Why Celery?**
- Robust task queue system
- Distributed worker support
- Retry and error handling
- Production-proven at scale

**Why PostgreSQL?**
- ACID compliance
- JSON support for flexible data
- Excellent performance
- Rich ecosystem
